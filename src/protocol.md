# 游戏网络通信协议文档

## 概述
这是一个基于 TCP 的实时游戏通信协议，用于在客户端和服务器之间同步游戏对象状态和玩家操作。

## 网络架构
- **传输层**: TCP
- **端口**: 8888
- **数据格式**: SFML Packet (二进制序列化)
- **通信模式**: 客户端-服务器模型

## 协议结构

### 基本数据包格式
所有数据包都使用 SFML Packet 进行序列化，包含类型标识和具体数据。

## 客户端到服务器的消息

### 玩家控制消息
当玩家执行操作时，客户端向服务器发送控制指令：

| 消息类型             | 数值 | 描述    |
|------------------|----|-------|
| Jump（按下W键）       | 0  | 跳跃操作  |
| Run Left（按下A键）   | 1  | 向左跑   |
| Run Right（按下D键）  | 2  | 向右跑   |
| Stop Left（释放A键）  | 3  | 停止向左跑 |
| Stop Right（释放D键） | 4  | 停止向右跑 |
| Stop Jump（释放W键）  | 5  | 停止跳跃  |

**示例**: `packet << 0` 表示玩家按下跳跃键

## 服务器到客户端的消息

### 消息类型定义
| 类型            | 数值 | 描述       |
|---------------|----|----------|
| Spawn Object  | 0  | 创建新游戏对象  |
| Update Object | 1  | 更新游戏对象状态 |
| Remove Object | 2  | 删除游戏对象   |

### 1. 创建对象消息 (Type: 0)

#### 对象类型定义
| 对象类型          | 数值 | 描述         |
|---------------|----|------------|
| Mario (自己)    | 0  | 当前玩家的马里奥角色 |
| Mario (其他玩家)  | 1  | 其他玩家的马里奥角色 |
| Player        | 2  | 圆形玩家对象     |
| BoxGameObject | 3  | 方块对象       |

#### 各对象数据结构

**Mario 对象 (类型 0 和 1)**:
```
<< 0 << obj_type << x << y << speed_x << speed_y << is_jumping << id
```

- `obj_type`: 0(自己) 或 1(其他玩家)
- `x, y`: 位置坐标
- `speed_x, speed_y`: 速度分量
- `is_jumping`: 是否正在跳跃 (bool)
- `id`: 对象唯一标识符

**Player 对象 (类型 2)**:
```
<< 0 << 2 << x << y << speed_x << speed_y << radius << id
```

- `radius`: 玩家半径

**BoxGameObject 对象 (类型 3)**:
```
<< 0 << 3 << x << y << speed_x << speed_y << width << height << id
```

- `width, height`: 方块尺寸

### 2. 更新对象消息 (Type: 1)

#### 对象更新类型
| 更新类型     | 数值 | 描述                      |
|----------|----|-------------------------|
| Mario 更新 | 0  | 马里奥角色状态更新               |
| 其他对象更新   | 1  | Player/BoxGameObject 更新 |

**Mario 更新**:
```
<< 1 << 0 << x << y << speed_x << speed_y << is_jumping << id
```


**其他对象更新**:
```
<< 1 << 1 << x << y << speed_x << speed_y << id
```


### 3. 删除对象消息 (Type: 2)
```
<< 2 << id
```

- `id`: 要删除的对象ID

## 连接流程

### 服务器启动
1. 监听端口 8888
2. 设置非阻塞模式
3. 等待客户端连接

### 客户端连接
1. 连接到指定服务器地址和端口 8888
2. 设置非阻塞模式
3. 开始接收初始化数据

### 新玩家加入流程
1. **客户端连接**: 新客户端连接到服务器
2. **场景同步**: 服务器发送当前所有游戏对象给新客户端
3. **创建新玩家**: 服务器为新客户端创建 Mario 角色
4. **广播通知**: 通知所有现有客户端有新玩家加入
5. **自我通知**: 向新客户端发送其自身信息

## 数据同步机制

### 服务器端同步
- **频率**: 每帧检查并处理客户端消息
- **内容**: 同步所有游戏对象的位置、速度和状态
- **优化**: 只发送变化的数据

### 客户端同步
- **接收处理**: 实时处理服务器发来的更新消息
- **对象管理**: 根据消息创建、更新或删除本地对象
- **状态维护**: 保持与服务器同步的游戏世界状态

## 错误处理

### 连接断开
- **客户端断开**: 服务器移除对应玩家并通知其他客户端
- **服务器断开**: 客户端设置网络状态为 None

### 数据验证
- 包完整性检查
- ID 有效性验证
- 数据范围检查

## 性能考虑

### 网络优化
- 使用非阻塞套接字避免线程阻塞
- 批量处理多个数据包
- 只同步必要的状态信息

### 带宽管理
- 位置和速度数据使用 float 类型
- 状态标志使用 bool 类型
- 对象 ID 使用 unsigned int

这个协议设计简单高效，适合实时游戏的状态同步需求。通过类型化的消息系统，可以轻松扩展支持更多游戏对象和操作类型。
